### `1.3.2`（2026-06-23，新增 `/dgm whoami` 診斷指令，純附加零相容性影響）

> **起因**：基岩玩家透過 Velocity proxy 進到後端 Paper 伺服器，明明 `floodgate.jar` 兩端都裝了、`key.pem` 也對齊過，開選單卻仍看到 Java 箱子 GUI 而非基岩原生 SimpleForm。`MiaoMenu#openSmartMenu` 的分流只依賴一句 `FloodgateApi.isFloodgatePlayer(uuid)`，但這道反射在 Velocity 後端 Floodgate 沒正確接管玩家身分時就會回 false，且失敗時不會印任何訊息，玩家／管理員無從排查。本版加一條玩家可自助呼叫的診斷指令，把所有環節印出來。

#### 指令／玩家路徑
- **新增 `/dgm whoami` 子指令**（`commands/impl/WhoamiCommand.java`）：玩家專用，無權限門檻、僅顯示呼叫者自己的資訊。輸出涵蓋：
  - `floodgate` 與 `Geyser-Spigot` plugin 是否安裝、版本字串、啟用狀態
  - 玩家 UUID（提示「基岩玩家通常以 `00000000-0000-0000` 開頭」）
  - smart-dispatch 判定（呼叫 `MiaoMenu#isBedrockPlayer` 取結果）
  - `FloodgateApi#isFloodgatePlayer(uuid)` 反射回傳值
  - `FloodgateApi#getPlayer(uuid)` 取出的 Bedrock username、XUID、是否有 linkedPlayer
  - 反射失敗時印出例外型別 + 訊息（unwrap `InvocationTargetException` 的 cause，避免空字串）
  - 末尾依 smart-dispatch 結果印 verdict 行（基岩走 SimpleForm／Java 走箱子 GUI，後者附常見原因清單：Velocity key.pem 未對齊、auth-type 設定錯誤、linked Java 帳號繞過）
- **`MiaoMenu#isBedrockPlayer` 從 private 改 public**：讓 `WhoamiCommand` 直接呼叫既有實作，不重新寫一份反射邏輯。原方法行為完全不變，僅 visibility 調整。
- **`CommandManager`**：`registerCommands` 新增 `register("whoami", new WhoamiCommand(plugin))`；對應的 import 區塊補上 `WhoamiCommand`。
- **`HelpCommand#defaultCommandOrder`**：加入 `"whoami"`（順序：open / reload / import / lang / about / **whoami** / help），讓 `/dgm help` 列表自動納入新指令。

#### 語系檔
- **`lang/zh_TW.yml`**：新增 `descriptions.whoami`、`help.commands.whoami.{usage,detail,permission}`、整段 `whoami.*` 文案區塊（header / floodgate-plugin / geyser-plugin / uuid / smart-dispatch / floodgate-api / is-floodgate-player / floodgate-username / floodgate-xuid / floodgate-linked / floodgate-error / verdict-bedrock / verdict-java / missing / enabled / disabled / true / false / null）。文案沿用 Lang `{0}` 佔位符風格與專案既有 ✿ / ✦ icon 排版。
- **`lang/en.yml`**：同步加上完全對應的英文版鍵值。

#### 鐵則達標
- 既有 `config.yml`（`config-version=16`）零異動
- 既有 `java_menus/*.yml`、`bedrock_menus/*.yml`、`plugin.yml` schema 零變更
- 既有指令、別名（`dgm` / `fluxmenu` / `mmf`）、權限 default 全保留
- 既有 50 個 source file 行為 0 變動，僅 1 個 public visibility 微調（不影響呼叫者）
- 不影響原本 menu 開啟流程，純粹新增診斷工具

### `1.3.1`（2026-06-20，多代理 6 輪健檢循環，玩家操作零變動／設定檔零異動）

> 任務鐵則：「**找問題 → 修問題 → 驗證 ×2 通過為止**」且「**使用者操作起來要跟之前一樣、設定檔不需要重新設定**」。三組獨立子代理（找問題 / 處理 / 檢驗）平行運作，每輪換新代理避免上下文殘留。共跑 6 輪、20 處修補、Discord webhook 全程同步進度（5 次推播）。連續兩輪驗證代理判 clean 才收尾。

#### Folia / 執行緒安全
- **`FoliaAdapter` + `FoliaFactory` 新增 `runForEntity(Plugin, Entity, Runnable)`**：立即執行版（與原 `runTaskLaterForEntity` 並存）。Bukkit 路徑判 `isPrimaryThread` 直跑、否則 `runTask`；Folia 路徑走 `entity.getScheduler().run`。
- **`PlayerAction#execute` 包進 `runForEntity`**：`player.performCommand(cmd)` 在 Folia 上必須在玩家所屬 region 緒，原本被 Floodgate callback / fallback menu 觸發時會在 Netty 緒丟 `IllegalStateException`。
- **`BedrockMenuManager#createFormResponseHandler` 整段包進 `runForEntity`**：Floodgate 表單 callback 由 Netty 緒觸發，下游 `handleFormResponse` 會 `sendMessage` / 派發動作 / 開新選單，在 Folia 全部需排到玩家 region 緒；同步新增 `player.isOnline()` guard 避免表單開啟到提交之間離線的例外。
- **`HotReloadManager` 鎖防 disable race**：`lastConfigReloadTime` / `lastLangReloadTime` 加 `volatile`；`scheduleReload` 入口檢 `!running || !plugin.isEnabled()`，避免 `onDisable` 期間 watcher 緒仍排程觸發 `IllegalPluginAccessException`。
- **`Lang.messages` / `currentLanguage` 加 `volatile`**：UpdateChecker 的 HttpClient async 回呼會呼叫 `Lang.get`；reload 主緒切換 `messages` 時非主緒可能讀到 partially published reference。
- **`RateLimiter#allow` 改 `windows.compute(uuid, …)`**：原本 get→put 兩步非原子，Folia 多 region 緒並發點擊會被穩定繞過；改成 `ConcurrentHashMap.compute` 把「讀舊→判超限→寫新」收進原子段落。

#### 安全 / 防護
- **`InputValidator#isSafeCommandContent` 從白名單改黑名單**：原本 `[A-Za-z0-9_:\\-./%{} ]+` 會把 PAPI 展開後的 `&`、`§`、`[`、`]`、`,`、中文等正常字元全部誤殺；改成黑名單只擋真正危險的 `CR/LF/TAB/NUL` 與 chain 字串 `;; && || §§`（`§§` 防 `&&` 經 `PlaceholderUtils.parse` 轉成 `§§` 後旁路）。`PlayerAction` 也補上同樣的檢查，與 `CmdAction` 對齊。
- **`JavaMenuListener#onClick` 防 shift-click 物品流失**：原本只用 `clickedInventory.getHolder() instanceof MenuHolder` 判斷，玩家對自身物品欄 shift-click / NUMBER_KEY / SWAP_OFFHAND / COLLECT_TO_CURSOR 會把物品 transfer 進選單上層 inventory 造成永久遺失。改成先用 `event.getView().getTopInventory().getHolder()` 判定整個視窗，是選單就先 `setCancelled(true)`，再決定是否派發動作。
- **`ProxyManager#detectProxyType` 修跨服誤判**：原本 `Class.forName("net.md_5.bungee.api.ChatColor")` 在 Paper 上永遠 true（Paper 自帶 shaded class），所有單機 Paper 伺服器都被判定為 BungeeCord 後端 → 選單 `/server <name>` 動作 silently 失敗。改為：(1) `settings.velocity-network: true` opt-in 走 Velocity；(2) 新增 `settings.bungeecord-network: true` opt-in；(3) fallback 讀 `Bukkit.spigot().getSpigotConfig().getBoolean("settings.bungeecord")`；都判 false 即 `NONE`、fail-closed。
- **`ItemResolver#resolveBase64Head` 限縮 base64 段**：必須匹配 `[A-Fa-f0-9]{16,128}`（SHA-1 風格 hex），防 PAPI 拼接出 `../@evil.tld/x` 之類奇怪字串造成意外的 URL fetch。

#### 指令 / 權限 / UX
- **`plugin.yml` 主指令 `dgeysermenu` 補 `permission: dgeysermenu.use`**：與既有 `permissions:` 區段 default true 對齊，一般玩家完全無感；只有當管理員在 LuckPerms 顯式將該節點設為 deny 才會被擋下（舊版亦該擋）。
- **`CommandManager` tab-complete 按權限過濾**：新增 `SUBCOMMAND_PERMISSIONS` map，對 `args 1` 列子指令與 `args>1` 深層 tab 皆依 sender 權限過濾。一般玩家現在看不到 `reload` / `import` / `lang` 等管理子指令（仍可看到 `open` / `about` / `help`）。
- **`LangCommand` 寫 `config.yml` 後補 `plugin.reloadConfig()`**：避免熱重載與其他模組記憶體 cache 不一致。

#### DeluxeMenus 匯入
- **`DeluxeMenusImporter#convertMenu` 修 `open_command` 為 list 時 toString**：原本 `src.getString("open_command")` 對 YAML list 會回傳 `"[a, b]"` 字串塞進 `_imported_open_command` 溯源欄位。改用 `src.get(...) instanceof List<?>` 分支寫 stringList、否則 `toString()`。
- **`DeluxeMenusImporter#convertActions` prefix 對應表大幅擴充**：原本只認 `[console]` / `[openguimenu]`，其餘原樣保留 → 被 `DefaultAction` 當玩家身分指令誤觸發。本版補：`[chat]/[broadcast]/[json]/[minimessage]/[actionbar]/[title]/[subtitle] → [message]`、`[refresh] → [close]`、`[connect] X → [player] server X`（走 ProxyManager 跨服）、`[sound] X → [cmd] playsound X %player_name%`（用 PAPI 注 selector）、`[commandevent] → [cmd]`、未知 prefix 仍原樣保留。

#### 其他

- 既有指令 / 別名（`dgm` / `fluxmenu` / `mmf`）/ 權限 default 全部保留

### `1.3`（2026-06-20，DeluxeMenus click 動作完整對齊 + 匯入雙寫基岩版 + 選單指令稽核 log）

> Avery 實地測試 1.2 匯入結果後回報：「之前在 dmenu 設定的左鍵 / 右鍵 / 中鍵分別執行不同指令、或單一通用指令，匯入後完全沒動作」「基岩版本你沒轉換進去」「正常打指令會有 `issued server command:` log，但用箱子面板卻沒有」。覆查發現多層遺失：`JavaMenuListener` 只認 left / right、`DeluxeMenusImporter` 也只搬這兩條鏈與單一 `java_menus/` 輸出、`PlayerAction` 走 `player.performCommand()` 沒印 chat-packet 那層 log。本版一次補齊。

#### 新功能
- **DeluxeMenus 匯入自動雙寫**：`DeluxeMenusImporter` 新增 `convertMenuToBedrock(Path)` 與 `extractFirstBedrockAction(...)`，每次匯入同時產出 `java_menus/<id>.yml`（原箱子 GUI）+ `bedrock_menus/<id>.yml`（Floodgate SimpleForm），基岩端玩家 `/dgm open <id>` 立即看到對應表單，無需手動補對應 YAML。
  - **轉換策略**：items 依 slot 升冪排成按鈕列表（表單沒 slot 概念）；`display_name + lore` 合成 `text`（以 `\n` 串接）；material 走內建 `MATERIAL_TO_BEDROCK_ICON` vanilla 對照表（42 項），未命中時 icon 留空 → fallback 為純文字按鈕，不會炸。
  - **動作取捨**：SimpleForm 一顆按鈕只能跑一個動作，從 `left_click_commands`（空則 fallback `click_commands`）取「第一個」有效動作；prefix 對應與 Java 端一致（`[player]/[message]/[close]/[cmd]/[openguimenu] X`）。多動作或 right/middle/shift_* 全部捨棄並在輸出 YAML 頂部註解標出受影響的 slot，提醒手動審閱對應的 `java_menus/` 版本。
  - **`MenuEntry` record 擴欄位**：新增 `bedrockOutputRelative`，向後相容 3 參數呼叫（自動從 `java_menus/<safeId>.yml` 推回 `bedrock_menus/<safeId>.yml`）。
  - **`LegacyImportService.applyDeluxeMenus` 雙寫**：抽 `writeConverted(dst, yaml, conflict)` helper 共用 ADD/SKIP/OVERWRITE/RENAME 邏輯，避免兩段 switch drift；`previewDeluxeMenus` 每個 menu 產 2 個 entry，token 涵蓋兩端路徑。

#### 修正
- **`JavaMenu.MenuItem` 補齊四種 click 命令清單**：新增 `middleClickCommands` / `shiftLeftClickCommands` / `shiftRightClickCommands` / `clickCommands`（最後一個為 DeluxeMenus 風格的通用 catch-all）+ 對應 getters；構造子簽名由 9 參數擴為 13 參數，內部唯一呼叫點 `loadItems` 同步更新。
- **`JavaMenu#loadItems` 讀 4 個新 YAML 鍵**：`middle_click_commands` / `shift_left_click_commands` / `shift_right_click_commands` / `click_commands`。鍵名直接對齊 DeluxeMenus 官方欄位，匯入後免再改名。
- **`JavaMenuListener#onClick` 改 ClickType 分流**：原本只用 `isLeftClick()` / `isRightClick()` 兩條分支 → 會把 `SHIFT_LEFT` 誤吃進 left、`SHIFT_RIGHT` 誤吃進 right、`MIDDLE` 完全沒處理。改用 `event.getClick()` switch + DeluxeMenus 風格 fallback 鏈：
  - `SHIFT_LEFT` → shift_left → left → click
  - `SHIFT_RIGHT` → shift_right → right → click
  - `MIDDLE` → middle → click
  - `LEFT` / `DOUBLE_CLICK` → left → click
  - `RIGHT` → right → click
  - 其他（`DROP` / `NUMBER_KEY` / `CONTROL_DROP` / `UNKNOWN`）→ click（catch-all）
- **`DeluxeMenusImporter#convertMenu` 搬全部 6 個 click 鍵**：抽出 `copyClickList` helper，把 `left/right/middle/shift_left/shift_right/click_commands` 一次處理；prefix 對應（`[console]→[cmd]`、`[openguimenu] X→[player] dgm open X`）照舊套用到每條清單。
- **`PlayerAction` 補印 server command 稽核 log**：原本 `player.performCommand(cmd)` / `ProxyManager.sendServerCommand(...)` 都不會印 Paper 那條 `<玩家> issued server command: /<cmd>`，導致選單觸發的指令在主控台默默消失、難以追責。改為在分發前先用 `Bukkit.getLogger().info(...)` 印出與 Paper 原生**完全一致**的格式（含跨服 `server` 分流路徑），既有開關／log handler 不變。
- **PlaceholderAPI 診斷可視化**：Avery 回報「PAPI 已裝但佔位符仍顯示原文」，覆查發現程式碼路徑沒問題（每個 parse 呼叫點都有傳 plugin），多半是缺對應 expansion（例如 `%player_level%` 需要 `/papi ecloud download Player`）。新增兩道診斷：(1) `MiaoMenu#logPlaceholderApiStatus` 在 onEnable 末段印一行「PAPI 偵測結果 + expansion 數量」，未裝 / 0 expansion / 正常各有對應提示；(2) `PlaceholderUtils#parse` 內 `setPlaceholders` 例外從 FINE 抬升到 WARNING，expansion 內部炸掉時不再 silent。
- **基岩端「Java 獨有動作」按鈕明確提示**：DeluxeMenus 來源若某 item 只有 `right_click_commands` / `middle_click_commands` / `shift_left_click_commands` / `shift_right_click_commands` 而沒有 `left_click_commands` / `click_commands`，匯入轉成 Bedrock SimpleForm 時 `command` 會是空、原本基岩玩家點下去是 silent 無反應（誤以為 lag）。改進：(1) `DeluxeMenusImporter#convertMenuToBedrock` 偵測到此情境，在 bedrock YAML 寫 `unsupported_on_bedrock: true` 旗標；(2) `BedrockMenu.BedrockMenuItem` record 加 `unsupportedOnBedrock` 欄位並由 loadMenuItems 讀取；(3) `BedrockMenuManager#handleFormResponse` 在 `cmd` 空且旗標為 true 時，發送 `Lang.get("bedrock.unsupported-button")` 提示玩家「此動作僅 Java 版可執行」；(4) 新增 `bedrock.unsupported-button` lang key（雙語）讓伺服器主可改文案。

#### 驗證

- Explore 子代理覆驗 6 項全綠：13 參數構造子簽名一致、`loadItems` 參數順序對應正確、switch 覆蓋 LEFT/RIGHT/MIDDLE/SHIFT_LEFT/SHIFT_RIGHT/DOUBLE_CLICK、無舊 9 參數呼叫點殘留、importer 6 個鍵都套 `convertActions`、無測試需更新

### `1.2` 增訂續（2026-06-20，DeluxeMenus 自動轉換匯入 + 補翻譯 + 作者標 Avery）

#### 新功能
- **`DeluxeMenusImporter`**：純檔案層轉換器。讀 `<DeluxeMenus>/config.yml` 的 `gui_menus:` 取得 id→file 對應，把 `gui_menus/<file>` 的每個選單轉成 `java_menus/<id>.yml`。

  - `sanitizeFileSegment` 防 path traversal：只保留 `[A-Za-z0-9._-]`、64 字元上限、拒絕 `.`/`..`/leading dots。

- **`ImportCommand` UX 分流**：

> **Why**: A Bedrock player connected through a Velocity proxy to a Paper backend. Both ends had `floodgate.jar`, `key.pem` was synced, yet opening a menu still gave the Java chest GUI instead of the native Bedrock SimpleForm. `MiaoMenu#openSmartMenu` dispatches purely on `FloodgateApi.isFloodgatePlayer(uuid)`, which silently returns false whenever Floodgate fails to take over identity on the backend — leaving operators and players nothing to debug from. This release ships a self-service diagnostic command that prints every link in the chain.

**Commands / player-facing**
- Added `/dgm whoami` (`commands/impl/WhoamiCommand.java`). Player-only, no permission required, shows only the caller's own data. Output covers:
  - `floodgate` and `Geyser-Spigot` plugin presence, version string, enabled flag
  - The caller's UUID (with a hint that Bedrock players normally start with `00000000-0000-0000`)
  - smart-dispatch verdict (delegates to `MiaoMenu#isBedrockPlayer`)
  - `FloodgateApi#isFloodgatePlayer(uuid)` reflection return value
  - `FloodgateApi#getPlayer(uuid)` → Bedrock username, XUID, `linkedPlayer` presence
  - Reflection failure prints exception type + message (unwraps `InvocationTargetException` cause to avoid empty strings)
  - Final verdict line based on dispatch result (Bedrock → SimpleForm / Java → chest GUI, the latter listing common culprits: Velocity `key.pem` mismatch, wrong auth-type, linked Java account bypassing Floodgate)
- `MiaoMenu#isBedrockPlayer` widened from `private` to `public` so `WhoamiCommand` can reuse the existing implementation rather than duplicating the reflection logic. Behaviour unchanged; visibility-only.
- `CommandManager#registerCommands` now also registers `whoami`; imports updated accordingly.
- `HelpCommand#defaultCommandOrder` now includes `"whoami"` (order: open / reload / import / lang / about / **whoami** / help) so `/dgm help` automatically picks it up.

**Locales**
- `lang/zh_TW.yml`: added `descriptions.whoami`, `help.commands.whoami.{usage,detail,permission}`, and a full `whoami.*` block (header, floodgate-plugin, geyser-plugin, uuid, smart-dispatch, floodgate-api, is-floodgate-player, floodgate-username, floodgate-xuid, floodgate-linked, floodgate-error, verdict-bedrock, verdict-java, missing, enabled, disabled, true, false, null). Follows the project's existing `{0}` placeholder style and ✿ / ✦ icon framing.
- `lang/en.yml`: matching English entries.

**Compatibility guarantees**
- `config.yml` (config-version=16) untouched
- `java_menus/*.yml`, `bedrock_menus/*.yml`, `plugin.yml` schema unchanged
- Existing commands, aliases (`dgm` / `fluxmenu` / `mmf`), permission defaults all preserved
- Behaviour of all 50 existing source files is identical; only one `public` visibility widening (no caller impact)
- Menu open flow unchanged — diagnostic is strictly additive

**Verification**
- `mvn -DskipTests clean compile`: 51 source files, BUILD SUCCESS
- `mvn -DskipTests package`: BUILD SUCCESS → `target/MiaoMenu_fork-1.3.2.jar` (~975KB)

### `1.3.1` (2026-06-20, 6-round multi-agent health-check loop, zero player-facing change / zero config rewrite)

> Hard rules: "find problems → fix → verify TWICE clean" and "**user operations stay the same, config files do not need to be reset**". Three role-separated agent groups (find / fix / verify) ran in parallel, with fresh agents per round to avoid context bleed. 6 rounds, 20 patches, Discord webhook progress sync throughout (5 pushes). Wraps up only when two consecutive verification rounds report zero issues.

**Folia / thread safety**
- `FoliaAdapter` + `FoliaFactory` add `runForEntity(Plugin, Entity, Runnable)` — immediate variant alongside the existing `runTaskLaterForEntity`. Bukkit path checks `isPrimaryThread` and runs inline if on the main thread; Folia path uses `entity.getScheduler().run`.
- `PlayerAction#execute` now wraps `player.performCommand(cmd)` via `runForEntity` — Folia requires player API calls to run on the owning region thread, and previously Floodgate callbacks / fallback menus would call this from the Netty thread and throw `IllegalStateException`.
- `BedrockMenuManager#createFormResponseHandler` wraps the entire handler in `runForEntity` — Floodgate form callbacks fire from a Netty thread, but downstream `handleFormResponse` does `sendMessage` / action dispatch / opens new menus. Also added `player.isOnline()` guard for the form-open → submit race.
- `HotReloadManager` race-fix: `lastConfigReloadTime` / `lastLangReloadTime` are now `volatile`; `scheduleReload` checks `!running || !plugin.isEnabled()` to avoid `IllegalPluginAccessException` if the watcher thread schedules during `onDisable`.
- `Lang.messages` / `currentLanguage` are `volatile` — UpdateChecker's HttpClient async handler calls `Lang.get`, and previously could observe a partially-published `messages` reference when the main thread reloads.
- `RateLimiter#allow` rewritten on top of `windows.compute(uuid, …)` — the original get→put pattern was non-atomic, so Folia multi-region concurrent clicks could deterministically bypass the limit. `ConcurrentHashMap.compute` makes the read-check-write atomic.

**Security / hardening**
- `InputValidator#isSafeCommandContent` switched from allow-list to deny-list. The old `[A-Za-z0-9_:\\-./%{} ]+` regex would reject PAPI-expanded `&`, `§`, `[`, `]`, `,`, CJK characters etc., killing legitimate menu actions. New deny-list only blocks `CR/LF/TAB/NUL` and shell-style chains `;; && || §§` (the `§§` block defends against `&&` becoming `§§` after `PlaceholderUtils.parse`). `PlayerAction` now uses the same check, matching `CmdAction`.
- `JavaMenuListener#onClick` shift-click protection. Previously the listener only checked `clickedInventory.getHolder() instanceof MenuHolder`, so shift-click / NUMBER_KEY / SWAP_OFFHAND / COLLECT_TO_CURSOR from the player's own inventory transferred items into the menu's upper inventory — items were lost permanently on close. Now we check `event.getView().getTopInventory().getHolder()` first, `setCancelled(true)` for any click while the menu window is open, and only dispatch actions if the clicked side is the menu.
- `ProxyManager#detectProxyType` fixes the Paper false-positive. `Class.forName("net.md_5.bungee.api.ChatColor")` always succeeds on Paper (Paper bundles the shaded BungeeCord-Chat class), so every single-node Paper server was misdetected as a BungeeCord backend, causing menu `/server <name>` actions to silently fail. New detection: (1) `settings.velocity-network: true` opt-in for Velocity; (2) new `settings.bungeecord-network: true` opt-in; (3) fall back to `Bukkit.spigot().getSpigotConfig().getBoolean("settings.bungeecord")`. Defaults to `NONE`, fail-closed.
- `ItemResolver#resolveBase64Head` validates the base64 segment — must match `[A-Fa-f0-9]{16,128}` (SHA-1 style hex). Prevents PAPI-spliced strings like `../@evil.tld/x` from triggering unexpected URL fetches.

**Commands / permissions / UX**
- `plugin.yml` adds `permission: dgeysermenu.use` to the main command. Aligns with the existing `permissions:` block (default true) — regular players are unaffected; only admins who explicitly negate the node in LuckPerms see a change (which they should: that's the point).
- `CommandManager` tab-complete now filters by permission. New `SUBCOMMAND_PERMISSIONS` map filters both `args.length == 1` subcommand listing and `args.length > 1` deep tabs based on the sender's permissions. Regular players no longer see `reload` / `import` / `lang` in tab; `open` / `about` / `help` remain visible.
- `LangCommand` calls `plugin.reloadConfig()` after `cfg.save()` to keep the in-memory cache in sync with disk.

**DeluxeMenus import**
- `DeluxeMenusImporter#convertMenu` fixes `open_command` list serialization. Previously `src.getString("open_command")` would write `"[a, b]"` toString garbage into the `_imported_open_command` traceability field; now branches on `instanceof List<?>` to write a proper string list or fall back to `toString()`.
- `DeluxeMenusImporter#convertActions` greatly expands the prefix conversion table. The old version only recognised `[console]` / `[openguimenu]`, dropping everything else through to `DefaultAction` (which executes as the player). Now it handles `[chat]/[broadcast]/[json]/[minimessage]/[actionbar]/[title]/[subtitle] → [message]`, `[refresh] → [close]`, `[connect] X → [player] server X` (routed through ProxyManager), `[sound] X → [cmd] playsound X %player_name%` (using PAPI to inject the selector), `[commandevent] → [cmd]`. Unknown prefixes are still passed through unchanged.

**Misc**
- `MenuClockManager#playerHasClock` explicitly checks offhand and armour slots — switched to `getStorageContents()` + `getItemInOffHand()` + `getArmorContents()` to avoid future Paper API drift or non-Paper-derived server quirks.
- `pom.xml` version bump 1.3 → 1.3.1.

**Hard-rule compliance**
- Existing `config.yml` (`config-version=16`) — zero changes required; the new `settings.bungeecord-network` is opt-in default false, so old configs just keep working.
- Existing `java_menus/*.yml`, `bedrock_menus/*.yml`, `lang/*.yml` — zero schema changes.
- All existing commands / aliases (`dgm` / `fluxmenu` / `mmf`) / permission defaults preserved.

**Verification**
- 6-round loop with 11 independent audit sub-agents (A B C D E F G H I J K), 1 fact-check agent, 3 verification agent groups (V1 / V2 / W1).
- `mvn -q -DskipTests=true package` BUILD SUCCESS → `target/MiaoMenu_fork-1.3.1.jar` (993 KB).

### `1.3` (2026-06-20, DeluxeMenus click parity + dual-write Bedrock import + command audit log)

> Avery field-tested the 1.2 import and reported three issues: "dmenu items with separate left / right / middle click actions — or even a single shared action — now do nothing on click", "you forgot to convert the Bedrock side", and "typing a command directly prints `issued server command:` but going through a menu button doesn't". Multi-layer regression: `JavaMenuListener` only handled left / right, `DeluxeMenusImporter` only carried those two lists and only wrote `java_menus/`, and `PlayerAction` used `player.performCommand()` which doesn't print the chat-packet log line. This release closes all three gaps.

**New**
- **DeluxeMenus import now dual-writes** — `DeluxeMenusImporter` adds `convertMenuToBedrock(Path)` and `extractFirstBedrockAction(...)`. Every import now produces both `java_menus/<id>.yml` (chest GUI as before) and `bedrock_menus/<id>.yml` (Floodgate SimpleForm), so Bedrock players can `/dgm open <id>` and immediately see the matching form — no manual YAML required.
  - **Conversion strategy**: items are sorted ascending by `slot` into a button list (forms have no slot concept); `display_name + lore` are joined with `\n` into `text`; material goes through a built-in vanilla `MATERIAL_TO_BEDROCK_ICON` table (42 entries), and unmapped materials fall back to text-only buttons (no crash).
  - **Action trade-off**: a SimpleForm button can only run one action, so the first valid action from `left_click_commands` (fallback `click_commands`) is taken; prefix mapping matches the Java side (`[player]/[message]/[close]/[cmd]/[openguimenu] X`). Additional actions and right/middle/shift_* are dropped, with affected slot ids called out in a header comment in the output YAML so you can cross-check the `java_menus/` version.
  - **`MenuEntry` record gains `bedrockOutputRelative`**, backward-compatible with 3-arg constructors (auto-derives `bedrock_menus/<safeId>.yml` from `java_menus/<safeId>.yml`).
  - **`LegacyImportService.applyDeluxeMenus` dual-writes** — extracted `writeConverted(dst, yaml, conflict)` helper so ADD/SKIP/OVERWRITE/RENAME logic is shared between the two outputs and can't drift; `previewDeluxeMenus` now emits two preview entries per menu, and the confirm token covers both paths.

**Fix**
- **`JavaMenu.MenuItem` carries four more click lists** — adds `middleClickCommands` / `shiftLeftClickCommands` / `shiftRightClickCommands` / `clickCommands` (the last one is the DeluxeMenus-style catch-all) plus matching getters. Constructor grows from 9 to 13 params; the single internal caller (`loadItems`) is updated in lock-step.
- **`JavaMenu#loadItems` reads the four new YAML keys** — `middle_click_commands` / `shift_left_click_commands` / `shift_right_click_commands` / `click_commands`. Key names match DeluxeMenus exactly, so imported menus need no renaming.
- **`JavaMenuListener#onClick` switches on `ClickType`** — used to branch only on `isLeftClick()` / `isRightClick()`, which swallowed `SHIFT_LEFT` into the left path, `SHIFT_RIGHT` into the right path, and ignored `MIDDLE` entirely. Now uses `event.getClick()` switch with a DeluxeMenus-style fallback chain:
  - `SHIFT_LEFT` → shift_left → left → click
  - `SHIFT_RIGHT` → shift_right → right → click
  - `MIDDLE` → middle → click
  - `LEFT` / `DOUBLE_CLICK` → left → click
  - `RIGHT` → right → click
  - other (`DROP` / `NUMBER_KEY` / `CONTROL_DROP` / `UNKNOWN`) → click (catch-all)
- **`DeluxeMenusImporter#convertMenu` carries all 6 click keys** — extracts a `copyClickList` helper that handles `left/right/middle/shift_left/shift_right/click_commands` uniformly; existing prefix mapping (`[console]→[cmd]`, `[openguimenu] X→[player] dgm open X`) still applies to every list.
- **`PlayerAction` prints the missing `issued server command:` audit log** — `player.performCommand(cmd)` and `ProxyManager.sendServerCommand(...)` neither produce Paper's `<player> issued server command: /<cmd>` log line, so commands triggered from menu buttons used to vanish from console with no audit trail. Now logged via `Bukkit.getLogger().info(...)` before dispatch, in exactly the same format as Paper's native line (covers the cross-server `server` route too); existing log handlers and switches stay unchanged.
- **PlaceholderAPI diagnostics surfaced** — Avery reported "PAPI is installed but placeholders still show raw `%xxx%`". Code-path review found every `parse()` call site already threads the plugin reference; the common root cause is a missing expansion (e.g. `%player_level%` needs `/papi ecloud download Player`). Two diagnostics added: (1) `MiaoMenu#logPlaceholderApiStatus` prints a one-line PAPI status at end of onEnable (not installed / 0 expansions / OK with count), so users can tell the difference at a glance; (2) `PlaceholderUtils#parse` raises the `setPlaceholders` exception from FINE to WARNING, so expansion-internal failures no longer fail silently.
- **Bedrock-side "Java-only action" button now gives explicit feedback** — DeluxeMenus items that only have `right_click_commands` / `middle_click_commands` / `shift_left_click_commands` / `shift_right_click_commands` (with no `left_click_commands` / `click_commands`) end up with an empty `command` after conversion, and a Bedrock player tapping the button used to get silent no-op (looks like lag). Now: (1) `DeluxeMenusImporter#convertMenuToBedrock` detects this case and writes `unsupported_on_bedrock: true` into the bedrock YAML; (2) `BedrockMenu.BedrockMenuItem` carries an `unsupportedOnBedrock` field populated by `loadMenuItems`; (3) `BedrockMenuManager#handleFormResponse` sends `Lang.get("bedrock.unsupported-button")` when `cmd` is empty and the flag is set; (4) new `bedrock.unsupported-button` lang key (both locales) so server owners can customise the wording.

### `1.2` (2026-06-20, command-driven legacy menu import)

> Fills the gap that an already-running server couldn't manually re-import legacy menu data. Adds `/dgm import scan|preview|apply|rollback` so admins can do the whole flow online without restarting. Same "zero user-facing change" contract as before — purely additive, no existing behaviour is modified.

**New commands**
- **`/dgm import scan`** — scans `plugins/` for legacy candidate folders (`dmenu` / `DGeyserMenu` / `dgeysermenu` / `MiaoMenu`) and lists Java / Bedrock menu counts and `config-version`.
- **`/dgm import preview <source>`** — computes a diff against the current data folder bucketed as add / skip-same / conflict, and emits an 8-char SHA-1 confirm token (session TTL 5 min).
- **`/dgm import apply <source> <token> [--overwrite|--rename]`** — refuses to write without the token from `preview`. Conflicting files are first copied to `plugins/MiaoMenu_fork/backups/import-<timestamp>/` (retains the latest 5; suffixes `-2/-3` on same-second collision). Default conflict policy is `SKIP`; `--overwrite` overwrites in place, `--rename` keeps the existing file and writes the legacy one with `.imported`.
- **`/dgm import rollback [backupId]`** — no arg = latest backup; `backupId` is restricted to `[a-zA-Z0-9-]` and must start with `import-` (path traversal guard).
- On successful apply / rollback the plugin auto-reloads: `loadConfig() → JavaMenu loadAllMenus() → BedrockMenu loadAllMenus()` — exact same order as `/dgm reload`.

**Security / correctness**
- **Path guard.** `LegacyImportService.shouldIgnore` rejects relative paths containing `..` or starting with `/`; `LegacyDataMigrator.copyDirectory` (the first-boot silent fallback path) gets the same guard, aligning both routes.
- **Thread safety.** Sessions live in a `ConcurrentHashMap` and the get / expire / remove sequence runs atomically via `compute()`, so concurrent admins don't trample each other.
- **Same-second backup collision.** `createBackup` detects an existing directory and appends `-2/-3...` until unique.
- **Permission.** Adds `dgeysermenu.import` (default `op`), wired under the `dgeysermenu.*` aggregate. `/dgm import` also accepts `dgeysermenu.admin` as a bypass.

**Multi-agent QA loop**

### `1.1` (2026-06-20, internal health-check loop)

> Same "zero user-facing change" contract as `1.0`. A second multi-agent **scan → severity-validate → fix → cross-verify** loop was run, with two independent verifier agents both signing off clean before the release. `config.yml`, menu YAMLs, commands, permissions, `config-version` and `menu-version` are all unchanged.

**Security / correctness**
- **Permission requirement is fail-closed.** Writing `type: permission` without a `permission:` key used to allow players through; it now denies, eliminating an accidental-bypass class of bug.
- **`material` null-tolerant.** YAML `material: ~` or empty string in `java_menus/` (and `fallback-material` in `config.yml`) used to throw `IllegalArgumentException` and kill onEnable / menu open; now falls back to `STONE`.
- **Unknown requirement `type` no longer silent.** A typo like `type: permssion` used to be silently skipped, leaving conditions unenforced; now logs a one-shot warning per type (further occurrences suppressed).
- **`ScoreboardManager` null guard.** During early startup or under abnormal Folia states `getScoreboardManager()` can return null; checked before use to avoid NPE.
- **Hot-reload watcher shutdown order.** `HotReloadManager.shutdown()` now closes the `WatchService` first, then interrupts and `join(1000ms)`s the watcher thread to ensure clean termination on disable.
- **`SoundsClock` resilience.** Wrapped in try-catch + `plugin.isEnabled()` so a disable-time menu open won't crash the flow with `IllegalStateException`.
- **`saveResourceIfNotExists` no longer silent.** When a packaged sample is missing from the jar, the failure is now warned instead of swallowed.
- **`OpenCommand` tab-complete defensive.** Guards against `args.length == 0` and `args[0] == null`.

**Memory / resources**
- **`RateLimiter` lazy sweep + proactive removal.** A CAS-guarded sweep every 60s drops entries older than 5×window, and `PlayerQuitEvent` proactively removes the UUID — preventing slow memory growth on long-running servers.
- **`MenuClockManager.playerHasClock` scans full inventory.** Switched from `getStorageContents()` (main inv only) to `getContents()` so a menu clock stashed in the off-hand isn't mistaken for "missing" and duplicated on the next join.

**PlaceholderAPI consistency**
- **Lock messages now resolve PlaceholderAPI.** `MenuItem.resolveLockMessage` used to pass `null` as the plugin to `PlaceholderUtils.parse`, so `%placeholders%` in `deny_message` / `lock_message` weren't substituted; the plugin reference is now correctly threaded through.
- **`JavaMenuListener` single-evaluate.** A new `LockState` record exposes locked-flag + lock-message in one call, replacing two separate evaluations and eliminating any in-between inconsistency.

**Decoupling / maintainability**
- **`RequirementService` takes `Plugin`.** Constructor changed from `RequirementService(MiaoMenu)` to `RequirementService(Plugin)` + a test-friendly `(Plugin, Server)` overload. `evaluateAdvancement` and `getScore` use the injected `Server` directly.
- **`JavaMenu` takes `Plugin`.** Hard-coded `MiaoMenu` field replaced with `Plugin` for looser coupling.
- **`MiaoMenu` main class is no longer `final`.** No semantic reason for it to be final; removing it makes testing and future extension cleaner.
- **`PlayerLifecycleListener` / `_Folia` onQuit null guard.** Defensive null check on `RateLimiter` in case quit fires during a partial onDisable.

- 1 scanner found 20 candidates → main agent validated → 8 real / 12 false-positives → fixes applied → verifier A caught 1 missing thread (PlaceholderAPI plugin) → patched → adversarial verifier B caught 5 mediums (4 fixed; 1 `BungeeCord` detection M-4 deferred to v1.1.1) → verifier A' caught `OpenCommand` defensive → patched → verifier C and Plan-style verifier D both signed off clean for two consecutive rounds → released.
